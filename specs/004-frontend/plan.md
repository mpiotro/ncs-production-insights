# 004 frontend — plan

Design for the **React + Vite + TypeScript** dashboard — the project's **visible surface**. A
single page: NCS fields on a **Leaflet** map (R1), select one to see its **oil-equivalents history +
24-month forecast** on a **Plotly** chart (R2) with the backtest **MAPE + credible/low-confidence**
badge (R3), and an explicit **"no credible forecast"** state for short-history fields (R4). All data
comes from the **003 API**, typed to its OpenAPI and **mockable by config alone** (R5); the frontend
**test + coverage** toolchain is established here (R6). Every element cites the EARS ID it serves
(principle 8). **Design only** — no component code, no tests (the seams below make both writable).

This phase is **purely additive and isolated**: it lives in a new top-level **`frontend/`** directory,
separate from the Python `src/ncs`. It touches no Python code; it consumes the *frozen* 003 API. The
TypeScript model types and the client interface are in `contracts.md`.

## 1. Project layout (under `frontend/`)

```
frontend/
  package.json            # scripts: dev | build | test | coverage | gen:api ; deps below
  package-lock.json       # committed (npm ci in CI — the JS analogue of uv.lock, R6)
  tsconfig.json           # strict: true (null-safety mirrors the | null contract)
  vite.config.ts          # Vite + React plugin + Vitest config (test+coverage block, R6)
  index.html              # Vite entry
  .env.example            # documents VITE_API_BASE_URL + VITE_API_SOURCE (no secrets, principle 7)
  src/
    main.tsx              # React root; reads config, picks the client (real|mock), mounts <App/>
    App.tsx               # layout shell: <FieldMap/> + <FieldDetailPanel/>; owns selectedNpdid (R1,R2)
    config.ts             # reads import.meta.env → { apiBaseUrl, apiSource } (R5)
    api/
      contracts.ts        # the friendly TS types + NcsApiClient interface (from contracts.md)
      schema.gen.ts       # GENERATED by openapi-typescript from openapi.json (R5 faithfulness)
      openapi.json        # committed snapshot of 003's /openapi.json (regen source)
      httpClient.ts       # real NcsApiClient — fetch against VITE_API_BASE_URL (R5 live)
      mockClient.ts       # mock NcsApiClient — in-memory fixtures, same interface (R5 mock)
      fixtures.ts         # mock data incl. a <60-month field with no forecast (makes R4 testable)
      selectClient.ts     # config → real|mock; the ONLY switch point (R5)
    components/
      FieldMap.tsx        # Leaflet GeoJSON layer; click a polygon → onSelect(npdid) (R1)
      FieldDetailPanel.tsx# orchestrates the selected field's fetches → chart + badge / empty (R2,R3,R4)
      ProductionForecastChart.tsx  # Plotly: history trace + forecast trace, visually distinct (R2)
      ForecastBadge.tsx   # MAPE % + credible/low-confidence indicator (R3)
      NoForecastNotice.tsx# explicit "no credible forecast available" message (R4)
    hooks/
      useFieldData.ts     # data-fetching hook: production + forecast for a npdid (loading/error/data)
    lib/
      productionSeries.ts # pure: ProductionHistoryResponse → {x[],y[]} oe series (null-aware) (R2)
      forecastSeries.ts   # pure: FieldForecast.points → {x[],y[]} (R2)
  test/
    setup.ts              # Vitest + Testing-Library/jest-dom setup
    acceptance/           # OWNED BY TEST-AUTHOR — black-box R1–R6 via mock client (principle 4)
    # unit specs live next to source as *.test.ts(x) — OWNED BY DEVELOPER (principle 4)
```

**Coexistence with the Python project + CI.** `frontend/` is a sibling of `src/` and `tests/`, with
its **own** package manager (npm) and config — it shares no tooling with `uv`/pytest. The Python CI
job is unchanged; a **separate Node job** runs the frontend (see §7). `frontend/node_modules`,
`dist/`, and coverage output are gitignored; `package-lock.json` and the `openapi.json` snapshot are
committed. The two stacks never import each other — the only contract between them is HTTP/OpenAPI.

## 2. Toolchain & libraries (with one-line justifications)

| Concern | Choice | Why (one line) |
|--------|--------|----------------|
| Build/dev server | **Vite + `@vitejs/plugin-react`** | tech-standards fixes Vite+React+TS; instant dev server, native ESM, first-class TS. |
| Language | **TypeScript, `strict: true`** | `strict` null-checks make the `\| null` contract (R2 nulls, R4 absent) a *compile* guarantee. |
| Test runner (R6) | **Vitest** | Vite-native (shares `vite.config.ts`/transforms — one config, no Babel/Jest duplication); Jest-compatible API. |
| Coverage (R6) | **`@vitest/coverage-v8`** | Vitest's official V8 coverage provider; `--coverage` emits text + lcov, the JS analogue of pytest-cov. |
| Component testing (R6) | **`@testing-library/react`** + **`jsdom`** + **`@testing-library/jest-dom`** | renders components in a DOM and asserts on *user-visible* output — the right grain for EARS acceptance tests. |
| Map (R1) | **Leaflet** + **`react-leaflet`** | tech-standards fixes Leaflet; `react-leaflet` is its idiomatic React binding (`<MapContainer>/<GeoJSON>/<TileLayer>`). |
| Map tiles (R1) | **OpenStreetMap raster tiles** via `<TileLayer>` | free, **no API token/paid tiles** (R1 + scope "out: paid tiles/tokens"); attribution shown. |
| Charts (R2) | **Plotly** (`plotly.js-dist-min`) + **`react-plotly.js`** | tech-standards fixes Plotly; `react-plotly.js` is the React wrapper; two traces give history-vs-forecast distinction. |
| Data fetching | **native `fetch`** in `httpClient.ts`, wrapped by a small **`useFieldData` hook** | zero-dependency; the typed client is the seam, so no data-library lock-in. (No TanStack Query — the page's data needs are small and a hook keeps the demo readable.) |
| Mock (R5) | **a second `NcsApiClient` impl** (`mockClient.ts`) returning `fixtures.ts`, chosen by `selectClient.ts` | mocking at the *client interface* (not at `fetch`) means app code is identical mock-vs-live; tests inject the mock directly. |
| Type faithfulness (R5) | **`openapi-typescript`** (dev-only) → `schema.gen.ts` from a committed `openapi.json` | regenerates model types from 003's real OpenAPI; drift becomes a reviewable diff, not silent rot. |

> **Why mock at the client interface, not at the network.** R5 demands "no change beyond
> configuration" between mock and live. A `fetch`/MSW-level mock would still exercise `httpClient`'s
> URL/parse code that has no business running in a unit test; swapping the **whole `NcsApiClient`**
> via `selectClient` keeps the boundary crisp and lets acceptance tests pass a mock client straight
> into components. (MSW is the documented alternative if true network-level integration tests are
> wanted later — see §8 Q3.)

## 3. Component tree & state (single-responsibility seams)

```
main.tsx
  └─ selectClient(config)  →  NcsApiClient        (real | mock — R5)
  └─ <App client=…>
        owns: selectedNpdid: number | null         (the one piece of cross-component state — R1→R2)
        ├─ <FieldMap                                 R1
        │     props: client, selectedNpdid, onSelect(npdid)
        │     loads /fields.geojson once, renders <GeoJSON> on OSM <TileLayer>;
        │     clicking a feature calls onSelect(feature.properties.field_npdid)
        └─ <FieldDetailPanel npdid=selectedNpdid client=…>     R2 R3 R4
              uses useFieldData(npdid) → { production, forecastResult, loading, error }
              ├─ if no selection → prompt "select a field"
              ├─ <ProductionForecastChart                       R2
              │     history = productionSeries(production)   (oe, null-aware)
              │     forecast = forecastSeries(forecast)      (only when a real FieldForecast)
              │     two visually-distinct Plotly traces (solid history vs dashed forecast)
              ├─ if forecastResult is FieldForecast → <ForecastBadge forecast/>   R3
              │     shows MAPE = round(backtest_mape*100)%  +  credible?"Credible":"Low confidence"
              └─ if forecastResult is ForecastNotAvailable → <NoForecastNotice/>  R4
                    explicit "No credible forecast available (insufficient history)"; chart still
                    shows history — never blank, never a fabricated curve
```

**State decisions.**
- **Single source of truth = `selectedNpdid` in `<App>`.** The map raises it (R1), the panel consumes
  it (R2). No global store (Redux/Zustand) — one lifted `useState` is all the page needs and keeps the
  seam obvious for the validator.
- **`useFieldData(npdid)`** isolates the two parallel fetches (`getProduction`, `getForecast`) and
  exposes `{ production, forecastResult, loading, error }`. It threads the **R4 typed outcome**
  through: `forecastResult` is `FieldForecast | ForecastNotAvailable | null`, so the panel branches on
  data, never on a caught exception.
- **Pure series builders** (`lib/productionSeries`, `lib/forecastSeries`) keep null-handling and the
  (year,month)→x-axis mapping out of the components — unit-testable in isolation (R2), and the place
  the JSON-`null` ≠ `0.0` rule is honored (a `null` oe month is a gap, not a zero).
- **Single-responsibility split** (validator checks this): map ≠ chart ≠ badge ≠ empty-state; data
  fetching (`useFieldData`) ≠ rendering; transport (`httpClient`) ≠ shape-building (`lib/*`).

## 4. API client seam (→ `contracts.md`, R5)

One interface, **`NcsApiClient`** (signatures in `contracts.md`), with two implementations chosen by
config **only**:

- **`httpClient.ts`** — real: `fetch` against `config.apiBaseUrl` (`VITE_API_BASE_URL`), parsing the
  exact JSON shapes. Endpoint URLs are the **un-prefixed** 003 paths verified against
  `src/ncs/api/routes/*` (`/fields`, `/fields/{npdid}`, `/fields/{npdid}/production`,
  `/fields/{npdid}/forecast`, `/fields.geojson`). 003's CORS default already allows the Vite origin
  `http://localhost:5173` (`src/ncs/api/settings.py`), so live integration is zero-config locally.
- **`mockClient.ts`** — mock: returns `fixtures.ts` in-memory data, **same interface**, no network.
  Fixtures include ≥1 credible field, ≥1 low-confidence field, and ≥1 field whose `getForecast`
  yields **`ForecastNotAvailable`** — so R3 and R4 are exercisable without live 003.
- **`selectClient.ts`** — the single switch: `config.apiSource === "mock" ? mockClient : httpClient`.
  **App/components never reference a concrete client** — they take `NcsApiClient`. This is what makes
  "mock ↔ live with no code change" (R5) structurally true.

**R4 surfacing (the load-bearing detail).** `getForecast` maps **404 `forecast_not_available`** to a
typed **`ForecastNotAvailable`** *value* (not a thrown error); every other non-2xx (incl. 404
`field_not_found`) throws. See `contracts.md` §"The R4 forecast outcome" for the discriminated union
and the rationale. The UI branches on the value (R4), so the insufficient-history state can never be
swallowed or rendered as blank/fabricated.

## 6. Testing strategy (seams only — author owns the tests)

Ownership per the constitution: **acceptance** (`frontend/test/acceptance/`) from the EARS criteria,
authored by the **test-author** (never the implementer, principle 4); **unit** specs (`*.test.ts(x)`
next to source) by the **developer**. Both run under Vitest with coverage (R6).

- **Acceptance (black-box, R1–R6)** — render real components with a **mock `NcsApiClient`** injected
  (the §4 seam), drive them via Testing-Library and assert user-visible output:
  - *R1*: render `<FieldMap>`; assert a GeoJSON layer with one path per fixture feature; simulate a
    polygon click → `onSelect` fires with that `field_npdid`.
  - *R2*: select a field → chart present with a **history** trace and a **forecast** trace, visually
    distinguishable (e.g. distinct trace `name`/dash); forecast trace has 24 points.
  - *R3*: a credible fixture shows MAPE % and a "Credible" badge; a low-confidence fixture shows "Low
    confidence" — matching `credible`/`backtest_mape`.
  - *R4*: the insufficient-history fixture shows the explicit "no credible forecast" notice **and**
    still renders the history trace, **no** forecast trace.
  - *R5*: same component test passes with the mock client; an additional test asserts `selectClient`
    returns the real `httpClient` when `apiSource !== "mock"` (config-only switch).
  - *R6*: satisfied by the suite running under `npm run coverage` and CI enforcing it.
- **Type-level conformance (R5 faithfulness)** — a `*.test-d.ts` using Vitest `expectTypeOf` asserts
  the friendly aliases (`Field`, `FieldForecast`, …) are assignable to the `schema.gen.ts` types, so
  drift from 003's OpenAPI fails the build (developer's unit-level task).
- **Unit (white-box, developer)** — `lib/productionSeries` (nulls → gaps, ordering), `lib/forecastSeries`,
  `httpClient` 404→outcome mapping (the R4 branch), `selectClient`.
- **Coverage reporting (R6)** — `@vitest/coverage-v8` emits text + lcov; CI fails under threshold. The
  **initial frontend coverage baseline is a coordinator call** (§8 Q1) — the *mechanism* is fixed here,
  the *number* is set when the suite first lands so the ratchet (principle 9) starts from a real value.

> Mockability is by **construction**: components depend on the `NcsApiClient` interface, so a test
> injects a mock with zero production-code change — the same property R5 requires for live↔mock.

## 7. CI — a Node job must be added (task for tasks.md)

`.github/workflows/ci.yml` is **Python-only** today (`uv sync` + `uv run pytest`; verified). The
frontend needs a **second, parallel job** — flag this as a `tasks.md` task:

```yaml
  frontend:
    runs-on: ubuntu-latest
    defaults: { run: { working-directory: frontend } }
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "20", cache: "npm", cache-dependency-path: frontend/package-lock.json }
      - run: npm ci
      - run: npm run coverage      # Vitest + v8 coverage; fails below the recorded baseline (R6, principle 9)
```

Notes for the task: keep the existing Python `test` job untouched (the two run in parallel); the Node
job is hermetic — it uses the **committed `openapi.json` snapshot** and the **mock client**, so it
needs **no live 003** and no network beyond the npm registry. A `tsc --noEmit` (or
`vitest --typecheck`) step enforces the `strict` null-safety contract.

## 8. Open questions

**Spec's three defaults — resolved (recommended):**
- **History streams** → **oe only.** Confirm the spec default: the chart shows the **oil-equivalents**
  history (matching the forecast target) only; the other five streams are out for this cycle. Keeping
  one stream makes history-vs-forecast the single visual story (R2). *Recommend: accept.*
- **Selection UX & layout** → **map-driven selection + a thin field list/search.** Click a polygon to
  select (R1→R2); also render a simple selectable list (from `listFields`) beside the map so a field
  with **null geometry** (no polygon on the map) is still selectable — otherwise such fields are
  unreachable. *Recommend: map click as primary, list as the accessible/null-geometry fallback.*
- **Test/coverage tooling** → **Vitest + `@vitest/coverage-v8` + Testing-Library/jsdom** (§2). The
  Vite-native runner avoids a second toolchain; V8 coverage is the official provider. *Recommend: accept.*

**New decisions I need from the coordinator (numbered; with my default):**
1. **Initial frontend coverage baseline (R6 / principle 9).** Python baseline is high (~97.6%); UI
   acceptance tests rarely hit that. *Recommend:* set the frontend ratchet's **first** baseline to the
   value the suite reports when it lands (record it in the PR), tracked **separately** from the Python
   baseline — they are different stacks and ratchets.
2. **Map tile provider.** OSM raster tiles are free/no-token but have an attribution + heavy-use
   policy. *Recommend:* OSM default tiles with attribution for the demo; revisit only if rate-limited.
3. **Network-level integration test of live 003.** The hermetic CI uses the mock client (no live 003).
   *Recommend:* keep CI mock-only (R5 is satisfied by the config-switch + the `selectClient` test); do
   a **manual** live-003 smoke run at integration time (roadmap: "integrates with 003 at the end"),
   not in CI. Add **MSW** only if we later want automated network-level tests.
4. **Plotly bundle size.** `react-plotly.js` pulls full Plotly (~3 MB). *Recommend:* use
   `plotly.js-dist-min` for the demo; defer a custom partial bundle (scatter-only) as a later
   optimization — out of scope for the demo's "it works and is visible" bar.
